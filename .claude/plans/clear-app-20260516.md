## Honest Takes

**Worth keeping as-is**
- Keeping Base/, Icon/, The/ components and lib/ intact avoids re-scaffolding generic UI when rebuilding. Correct call.
- Keeping toast + theme wired in main.ts — both are domain-agnostic and annoying to re-add later.

**Worth pushing back on**
1. **The/ components are handled reactively (fix-on-tsc-error)** — we don't know their contents until tsc fails. If TheSidebar has a complex menu tree with 10 apartment route names, the "stub to `[]`" instruction is ambiguous. Alternative: read TheSidebar and TheTopbar now before handing off, and give the executor explicit before/after diffs. Slightly more upfront work, zero ambiguity. Leaving as reactive for now — acceptable for a clear/reset task.

**Known debt this plan accepts**
- `auto-imports.d.ts` / `components.d.ts` will be stale until next `vite dev` — IDE may show phantom completions for deleted symbols until then. Harmless; regenerates on first dev server start.
- Auth starts from zero — first feature that needs login will require a new store + router guard from scratch.
